**Division of responsibilities.**

- **`core.py`** does not depend on any graphics libraries — pure algorithms importable anywhere. The implementations "from the lecture notes" are preserved verbatim: `knapsack_recursive` (brute force by recursion), `knapsack_brute_force` (explicit subset enumeration), `knapsack_greedy` + the `Item` class (greedy), `knapsack_dp` (the DP table). Next to them are the instrumented versions with step journals (`brute_force_steps`, `greedy_steps`, `knapsack_dp_steps`) and the backward-pass set reconstruction (`reconstruct_items` / `reconstruct_steps`) — the analogue of path reconstruction in graph algorithms.
- **`style.py`** — the single source of the visual theme: the color palette as named constants, `FIGURE_DPI`, fonts and `configure_style()`. The look of every figure can be changed from one place.
- **`i18n.py`** — bilingual labels: `t()` returns a string in the current language (`uk` by default / `en`), `set_lang()` switches it. The dictionary key is the Ukrainian string itself, so the Ukrainian figures stay byte-for-byte unchanged. Every visible label in `visualization.py`, `walkthrough.py` and the examples is wrapped in `t()`; completeness is guarded by the `missing_translations` audit in the smoke tests.
- **`visualization.py`** (needs `matplotlib`) takes the palette from `style.py` and draws everything static: the item showcase, the subset enumeration table, the recursion tree, the 2ⁿ growth chart, the DP table with its highlights (current cell, "take"/"skip" sources, the reconstruction path), the per-cell detail frames with the transition formula, and the knapsack packing strips. The console journals live here too (`print_dp_log`, `print_dp_table`, `print_subsets`).
- **`animation.py`** stitches frames into a looped **GIF** (via Pillow, always) and additionally encodes the same frame sequence into **MP4** (via `ffmpeg`, when available). GIF is the guaranteed inline-friendly format; MP4 is the lighter file with playback controls.
- **`walkthrough.py`** builds the step-by-step code ↔ table panels: the step journal (`build_overview_steps` / `build_cell_steps`) with a code-line highlight map, the code panel (`draw_code_panel`) and the grid compositor (`draw_code_walkthrough_grid`); the right panel reuses `draw_dp_table`. The color of a code line encodes the branch that fired (active / "take" won / value from above).
- **`examples/_common.py`** removes duplication between the examples: `Agg`/path setup, `save_figure`, `save_anim`, the CLI language switch and the `KnapsackExample` dataclass that keeps weights, values and capacity together.
- **`examples/_items.py`** — the source of truth for the three teaching instances (`CLASSIC`, `SMALL`, `BIG`); every example imports them, so within the examples the numbers are defined exactly once. The tests deliberately keep their own copies of these numbers (so that `test_core.py` stays free of heavy dependencies), and a smoke test guards that the copies match `_items.py`.
- The **`examples/`** scripts merely pick an instance from `_items.py`, call the algorithm and the drawing functions, and save the figures.
- **`knapSack.ipynb`** — the lecture-notes notebook the basic implementations and the teaching instances come from; it sits in the root as the primary source of this walkthrough.
